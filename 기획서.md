# claude-end-notification — 기획서

> Claude Code가 **작업을 끝냈을 때**와 **사용자에게 질문/권한을 요청할 때**
> macOS·Windows 시스템 알림을 띄워주는 Claude Code 플러그인.

작성일: 2026-05-14

---

## 1. 개요

Claude Code로 긴 작업을 돌려놓고 다른 창으로 넘어가면, 작업이 끝났는지·뭔가
물어보는지 알 수 없어 계속 터미널을 확인하게 된다. 이 플러그인은 두 순간에
OS 네이티브 알림을 띄워 "돌아와야 할 때"를 알려준다.

- **작업 완료** — Claude가 한 턴의 응답/작업을 마쳤을 때 (`Stop` 훅)
- **질문·권한 요청** — Claude가 권한을 묻거나 입력 대기로 멈췄을 때 (`Notification` 훅)

배포 형태는 **Claude Code 플러그인**. `/plugin install`로 설치하고,
마켓플레이스로 공유할 수 있다.

---

## 2. 목표 / 비목표

### 목표
- macOS·Windows에서 **추가 설치 없이** 동작 (OS 기본 알림 메커니즘 사용)
- 플러그인 하나로 두 이벤트(`Stop`, `Notification`) 모두 커버
- 설정 파일로 이벤트별 on/off, 소리, 노이즈 필터 조정
- 알림 본문에 **프로젝트명 + 무슨 상황인지**를 담아 한눈에 파악 가능

### 비목표 (v1 기준)
- 모바일/원격 푸시 알림 (별도 PushNotification 영역)
- Linux는 베스트에포트(`notify-send`)만, 1차 지원 대상 아님
- 작업 내용 요약·전송 등 트랜스크립트 분석 기능

## 3. 동작 시나리오

| # | 상황 | 트리거 | 알림 예시 |
|---|------|--------|-----------|
| 1 | 리팩터링을 시켜놓고 브라우저로 전환, 5분 뒤 완료 | `Stop` | **✅ Claude Code · my-app** — 작업을 마쳤어요 |
| 2 | 작업 중 `rm` 명령 권한을 물어봄 | `Notification` (`permission_prompt`) | **🔐 Claude Code · my-app** — Bash 실행 권한을 기다리는 중 |
| 3 | Claude가 질문을 던지고 입력 대기로 멈춤 | `Notification` (`idle_prompt` / `elicitation_dialog`) | **💬 Claude Code · my-app** — 입력을 기다리고 있어요 |
| 4 | 컨텍스트 한도로 턴이 잘림 | `Stop` (`stop_reason: max_tokens`) | **⚠️ Claude Code · my-app** — 토큰 한도로 응답이 중단됐어요 |

> ⚠️ `Stop` 훅은 **턴이 끝날 때마다** 발생한다. 즉 짧은 대화에서도 매 응답마다
> 알림이 뜬다. 이 노이즈 문제는 [7. 노이즈 제어](#7-노이즈-제어)에서 다룬다.

## 4. 기술 구조

### 4.1 플러그인 디렉터리 구조

```
claude-end-notification/
├── .claude-plugin/
│   └── plugin.json          # 플러그인 메타데이터
├── hooks/
│   └── hooks.json           # Stop / Notification 훅 등록
├── scripts/
│   └── notify.js            # 크로스플랫폼 알림 디스패처 (Node)
├── assets/
│   └── icon.png             # 알림 아이콘 (terminal-notifier/BurntToast용)
├── config.example.json      # 기본 설정 템플릿
└── README.md
```

Claude Code는 Node 런타임 위에서 동작하므로 `node`는 항상 사용 가능하다.
OS 분기(`process.platform`)를 셸 스크립트로 나누지 않고 **`notify.js` 한 파일**에서
처리해 macOS/Windows 코드 경로를 일원화한다.

### 4.2 `.claude-plugin/plugin.json`

```json
{
  "name": "claude-end-notification",
  "description": "Claude 작업 완료·질문 시 macOS/Windows 시스템 알림",
  "version": "0.1.0",
  "author": { "name": "hano", "email": "dev@keumbang.co.kr" },
  "license": "MIT",
  "keywords": ["notification", "hooks", "macos", "windows", "productivity"],
  "hooks": "./hooks/hooks.json"
}
```

### 4.3 `hooks/hooks.json`

훅 커맨드는 어느 환경에서나 검증된 **셸 문자열 형식**을 사용한다. 경로를
큰따옴표로 감싸 공백·인용부호 문제를 피하고, `${CLAUDE_PLUGIN_ROOT}`(플러그인
설치 경로)는 Claude Code가 실행 전에 치환한다. 이벤트 종류는 스크립트 인자로
넘긴다(`stop` / `notification`).

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node \"${CLAUDE_PLUGIN_ROOT}/scripts/notify.js\" stop"
          }
        ]
      }
    ],
    "Notification": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node \"${CLAUDE_PLUGIN_ROOT}/scripts/notify.js\" notification"
          }
        ]
      }
    ]
  }
}
```

- `Notification` 훅은 `matcher`로 알림 종류(`permission_prompt`, `idle_prompt`,
  `auth_success`, `elicitation_dialog` 등)를 거를 수 있지만, v1은 **전부 받아서
  `notify.js` 안에서 설정 기반으로 필터링**한다 (설정 한 곳에서 관리하기 위함).

### 4.4 `notify.js` 처리 흐름

```
stdin(JSON) ──┐
argv[2]       ├─▶ 1. 이벤트 페이로드 파싱
("stop"|      │   2. 설정 로드 (${CLAUDE_PLUGIN_DATA}/config.json, 없으면 기본값)
 "notification")│  3. 이 이벤트가 비활성? → exit 0
              │   4. 노이즈 필터(포커스/쿨다운) 통과 못함? → exit 0
              │   5. {title, body, sound} 메시지 빌드
              └─▶ 6. OS별 백엔드로 디스패치
                  7. 항상 exit 0  (Notification 훅은 결정권 없음,
                                    Stop 훅도 절대 block 하지 않음)
```

**핵심 입력 필드 (claude-code-guide 확인)**

`Stop` 훅 stdin:
```json
{
  "session_id": "abc123",
  "transcript_path": "/Users/.../transcript.jsonl",
  "cwd": "/Users/hano/dev/my/my-app",
  "permission_mode": "default",
  "effort": { "level": "medium" },
  "hook_event_name": "Stop",
  "stop_reason": "end_turn"          // end_turn | max_tokens | stop_sequence | tool_use
}
```

`Notification` 훅 stdin:
```json
{
  "session_id": "abc123",
  "transcript_path": "/Users/.../transcript.jsonl",
  "cwd": "/Users/hano/dev/my/my-app",
  "permission_mode": "default",
  "hook_event_name": "Notification",
  "notification_type": "permission_prompt",   // permission_prompt | idle_prompt
                                              // | auth_success | elicitation_dialog
                                              // | elicitation_complete | elicitation_response
  "notification_data": { "tool_name": "Bash", "tool_input": { "command": "rm -rf /tmp" } }
}
```

- 프로젝트명은 `cwd`의 basename으로 뽑는다.
- `notification_data.tool_name`이 있으면 본문에 도구명을 넣고, 없으면
  `notification_type`별 기본 문구로 폴백한다 (스키마가 타입마다 다르므로 방어적으로).

### 4.5 OS별 알림 백엔드

| OS | 기본 (설치 불필요) | 선택 (있으면 우선 사용) |
|----|--------------------|--------------------------|
| **macOS** | `osascript -e 'display notification …'` | `terminal-notifier` — 아이콘·클릭 동작·그룹화 지원 |
| **Windows** | PowerShell 네이티브 토스트 (`Windows.UI.Notifications`) | `BurntToast` 모듈 — 간결한 토스트 API |
| **Linux** | `notify-send` (베스트에포트) | — |

`config.macBackend` / `winBackend`를 `"auto"`로 두면 선택 도구가 PATH에 있을 때
자동으로 그쪽을 쓰고, 없으면 기본 백엔드로 폴백한다.

> **macOS 권한 주의:** `osascript` 알림은 "Script Editor" 이름으로 표시되며,
> 사용자가 *시스템 설정 → 알림*에서 한 번 허용해야 한다. `terminal-notifier`를
> 쓰면 "terminal-notifier" 이름으로 뜬다. 이 내용은 README에 명시한다.
> (terminal-notifier는 MIT라 플러그인에 동봉 배포도 검토 가능 — [10. 열린 질문](#10-열린-질문))

## 5. 이벤트별 알림 명세

| 이벤트 / 타입 | 아이콘 | 제목 | 본문 |
|---------------|--------|------|------|
| `Stop` (end_turn / tool_use) | ✅ | Claude Code · {project} | 작업을 마쳤어요 |
| `Stop` (max_tokens) | ⚠️ | Claude Code · {project} | 토큰 한도로 응답이 중단됐어요 |
| `Notification` permission_prompt | 🔐 | Claude Code · {project} | {tool_name} 실행 권한을 기다리는 중 |
| `Notification` idle_prompt | 💬 | Claude Code · {project} | 입력을 기다리고 있어요 |
| `Notification` elicitation_dialog | ❓ | Claude Code · {project} | Claude가 질문을 했어요 |
| `Notification` auth_success | 🔑 | Claude Code · {project} | 인증이 완료됐어요 (기본 off) |

- 제목에 `{project}`를 넣어 **여러 세션 동시 실행 시 어느 작업인지 구분**되게 한다.
- 알림 클릭 동작은 v2에서 (터미널 포커스 등) — [10. 열린 질문](#10-열린-질문).

## 6. 설정

`${CLAUDE_PLUGIN_DATA}/config.json` (플러그인 업데이트 후에도 유지되는 디렉터리).
없으면 `config.example.json`의 기본값을 사용한다.

```json
{
  "enabled": true,
  "events": {
    "stop": true,
    "notification": true
  },
  "notificationTypes": {
    "permission_prompt": true,
    "idle_prompt": true,
    "elicitation_dialog": true,
    "auth_success": false
  },
  "sound": "default",
  "skipWhenFocused": false,
  "focusedApps": ["Terminal", "iTerm2", "WezTerm", "Ghostty", "Code"],
  "minDurationSeconds": 0,
  "cooldownSeconds": 0,
  "macBackend": "auto",
  "winBackend": "auto"
}
```

| 키 | 설명 |
|----|------|
| `enabled` | 전체 on/off 마스터 스위치 |
| `events` | `Stop` / `Notification` 이벤트별 on/off |
| `notificationTypes` | `Notification` 세부 타입별 on/off |
| `sound` | 알림음 이름. `"none"`이면 무음 |
| `skipWhenFocused` | 터미널이 최전면일 때 알림 생략 ([7. 노이즈 제어](#7-노이즈-제어)) |
| `focusedApps` | "터미널 앱"으로 간주할 앱 이름 목록 |
| `minDurationSeconds` | 이 시간 이상 걸린 턴만 알림 (0 = 항상) |
| `cooldownSeconds` | 직전 알림 후 이 시간 내 재알림 억제 (0 = 끔) |

## 7. 노이즈 제어

`Stop` 훅은 **매 턴 종료 시 발생**하므로, 그대로 두면 짧은 대화에서 알림이
쏟아진다. 사용자가 원한 동작은 "프로세스가 끝났을 때"이지 "한 줄 답할 때마다"가
아니므로, 다음 필터를 둔다 (기본값은 보수적으로 — 동작을 막지 않음).

1. **포커스 감지 `skipWhenFocused` (권장, M3)**
   터미널 창을 보고 있으면 알림이 불필요하다. 최전면 앱을 조회해
   `focusedApps`에 속하면 알림을 생략한다.
   - macOS: `osascript`로 `System Events`의 frontmost 프로세스 조회 → 우선 구현
   - Windows: PowerShell로 foreground window의 프로세스명 조회 → 이어서 구현

2. **쿨다운 `cooldownSeconds`**
   `${CLAUDE_PLUGIN_DATA}`에 마지막 알림 시각을 저장해, 짧은 간격의 연속 알림을
   하나로 합친다.

3. **최소 소요시간 `minDurationSeconds`**
   "오래 걸린 작업만" 알리고 싶을 때. 턴 길이는 `Stop` 페이로드에 없으므로,
   `UserPromptSubmit` 훅을 하나 더 추가해 `${CLAUDE_PLUGIN_DATA}/session-<id>.start`에
   시작 시각을 기록하고 `Stop`에서 읽어 계산한다.
   → **선택 기능(M3)**, 기본값 0(끔).

> 설계 결정: v1 기본값은 "모든 `Stop`·`Notification`을 알림"이되, 위 세 필터를
> **설정으로 켤 수 있게** 제공한다. 포커스 감지를 켜는 것을 README에서 권장한다.

## 8. 배포 — 마켓플레이스

`.claude-plugin/marketplace.json`을 둔 git 저장소로 배포한다.

```json
{
  "name": "hano-tools",
  "owner": { "name": "hano", "email": "dev@keumbang.co.kr" },
  "description": "Claude Code 생산성 플러그인",
  "plugins": [
    {
      "name": "claude-end-notification",
      "source": "./",
      "description": "작업 완료·질문 시 시스템 알림",
      "version": "0.1.0"
    }
  ]
}
```

설치 흐름:
```
/plugin marketplace add hano/claude-end-notification
/plugin install claude-end-notification@hano-tools
```

저장소 자체를 마켓플레이스로 쓰면(루트에 `marketplace.json` + 플러그인 파일)
`source: "./"`로 단일 저장소 배포가 가능하다.

## 9. 엣지 케이스 & 리스크

| 항목 | 대응 |
|------|------|
| `Stop` 훅이 매 턴 발생 → 알림 폭주 | [7. 노이즈 제어](#7-노이즈-제어) 필터, README에서 포커스 감지 권장 |
| macOS 알림 권한 미허용 시 조용히 실패 | README에 *시스템 설정 → 알림* 허용 절차 명시, 첫 실행 안내 |
| `notification_data` 스키마가 타입마다 다름 | `tool_name` 등 옵셔널 접근 + 타입별 기본 문구 폴백 |
| 훅이 느리면 Claude 응답 체감 지연 | `notify.js`는 알림 명령을 **detach**하고 즉시 `exit 0` |
| 알림 명령 실패(도구 없음 등) | try/catch로 흡수, 절대 비정상 종료코드 반환 안 함 |
| 여러 세션 동시 실행 | 제목에 `{project}` 표기, 쿨다운/시작시각 파일은 `session_id`로 분리 |
| Windows PowerShell 실행 정책 | `-ExecutionPolicy Bypass`로 인라인 스크립트 실행 |

## 10. 열린 질문

1. **알림 클릭 동작** — 클릭 시 해당 터미널/세션을 포커스할까, 아무 동작 없이 둘까?
   (`terminal-notifier`/`BurntToast`는 클릭 액션 지원)
2. **terminal-notifier 동봉** — MIT 라이선스라 플러그인에 바이너리를 동봉해
   "설치 불필요 + 좋은 UX"를 동시에 줄 수 있다. 저장소 용량 vs UX 트레이드오프.
3. **Linux 지원 범위** — `notify-send`만으로 1차 포함할지, 아예 v2로 미룰지.
4. **소리 기본값** — 기본 `"default"`(소리 있음) vs `"none"`(무음)?
